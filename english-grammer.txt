--------------------------------
1) CAN
--------------------------------
Meaning:
Ability | Permission | Possibility | Informal request

USES:

A) Ability (present)
   - What someone is able to do now

   Example:
   I can write APIs.
   He can debug fast.

   Structure:
   S + can + V1 + O

B) Permission (informal)
   - Asking or giving permission

   Example:
   Can I leave early?
   You can use my laptop.

C) Possibility (general truth)
   - Something that is possible

   Example:
   This bug can cause failure.
   Too many joins can slow queries.

D) Informal request
   - Casual asking

   Example:
   Can you review my code?

Negative:
   cannot / can't

--------------------------------
2) COULD
--------------------------------
Meaning:
Past ability | Polite request | Weak possibility | Suggestion

USES:

A) Past ability
   - Ability in the past

   Example:
   I could run faster earlier.
   I could understand this code before.

B) Polite request (formal)
   - Softer than "can"

   Example:
   Could you please check this PR?
   Could you help me?

C) Weak possibility
   - Not 100% sure

   Example:
   This could be a DB issue.
   The server could be down.

D) Suggestion
   - Giving options

   Example:
   We could refactor this logic.

--------------------------------
3) WILL
--------------------------------
Meaning:
Future | Promise | Instant decision | Certainty

USES:

A) Future action
   - Decision made now

   Example:
   I will deploy tonight.
   We will fix this.

B) Promise / assurance
   Example:
   I will take care of it.

C) Instant decision
   - Decided at moment of speaking

   Example:
   Oh, I will handle this ticket.

D) Strong certainty
   Example:
   This will work.
   He will succeed.

Negative:
   will not ‚Üí won't

--------------------------------
4) WOULD
--------------------------------
Meaning:
Polite | Imaginary | Conditional | Past habit

USES:

A) Polite request (more formal)
   - More polite than "could"

   Example:
   Would you mind reviewing this?
   Would you like coffee?

B) Imaginary situation
   - Not real now

   Example:
   I would buy a Mac if I had money.
   I would refactor if I had time.

C) Conditional sentences
   Structure:
   If + past, would + V1

   Example:
   If I were you, I would fix this.

D) Past habit
   Example:
   He would write tests daily.

--------------------------------
5) SHALL
--------------------------------
Meaning:
Suggestion | Formal rule | Legal obligation

USES:

A) Suggestion (mostly with I / we)
   Example:
   Shall we start the meeting?
   Shall I create the ticket?

B) Formal / legal rule
   Example:
   The system shall log all errors.
   Employees shall follow policies.

Note:
- Rare in daily conversation
- Common in documents & contracts

--------------------------------
6) SHOULD
--------------------------------
Meaning:
Advice | Expectation | Duty | Probability

USES:

A) Advice / recommendation
   Example:
   You should write unit tests.
   We should optimize this query.

B) Expectation
   Example:
   The API should respond in 200ms.

C) Moral duty
   Example:
   We should follow coding standards.

D) Probability (logical)
   Example:
   He should arrive by now.
